IFT 2255

Devoir 3
Implémentation & Tests

Ce devoir se concentre sur les workflows d'implémentation, de test et de livraison. Le but est de mettre en pratique l'implémentation de votre design, effectuer une vérification du système programmé et livrer le produit Robotix.

Robotix

Dans un monde où la robotique est un domaine en constante évolution, les robots sont de plus en plus utilisés dans divers secteurs tels que l'industrie, la santé, l'agriculture et bien d'autres encore.

Avec l'augmentation de l'utilisation des robots dans les industries et même les résidences, la nécessité d'un système de gestion efficace pour ces machines devient de plus en plus pressante.

Exigences

Dans l'élaboration de votre solution, nous vous demandons de bien vouloir respecter les spécifications suivantes.
Pour l'implémentation, nous vous demandons minimalement une interface est en ligne de commande.

Fonctionnalités

  • Créer un compte
  • Se connecter
  • Modifier son profil
  • Enregistrer un robot
  • Supprimer un robot
  • Afficher l'état de mes robots
    i Vue général: Afficher le nom, le type, le niveau de batterie
    i Vue complète: Afficher en plus son numéro de série, sa position, sa vitesse, sa consommation CPU et mémoire.
  • Trouver une composante
    i Liste/Recherche: Afficher uniquement le nom et le type de composantes.
    i Fiche: Afficher toutes les informations
  • Acheter une composante
  • Trouver un fournisseur
    i Liste/Recherche: Afficher uniquement le nom et les types de composantes.
    i Fiche: Afficher toutes les informations (publiques)
  • S'inscrire ou se désinscrire à une activité
  • Voir ses notifications
  • Voir ses composantes
  • Enregistrer une composante
  • Modifier une composante
  • Supprimer une composante

Utilisateurs

  • Utilisateur Utilisateur principal de Robotix. Un compte distinct est associé à chaque utilisateur.
  • Fournisseur Utilisateur secondaire de Robotix gérant les composantes des robots. Un compte distinct est associé à chaque fournisseur.

Notes

  • Inscription L'adresse courriel et le pseudo doivent être uniques. Lors de la création d'un compte, un lien de confirmation (expirant après 24 heures) est envoyé à l'utilisateur. Passé le délai de confirmation, l'inscription est annulée.
  • Authentification Lors de l'authentification, un utilisateur fournit son adresse courriel et son mot de passe. Si l'authentification réussit, l'utilisateur est amené à l'écran d'accueil propre à son type de compte.
  • Trouver un fournisseur La recherche de fournisseur peut se faire, au moins, par nom et par types de composantes. Les accents ne sont pas pris en considération.
  • Trouver une composante La recherche de composantes peut se faire, au moins, par nom, par type ou par fournisseur. Les accents ne sont pas pris en considération.
  • Achat de composantes Lorsqu'un client achète une composante du fournisseur, le fournisseur reçoit une notification et son inventaire est mis à jour.
  • Notification Un utilisateur peut choisir de recevoir ces notifications par courriel.

Données

L'application devrait être initialisée avec la configuration minimale suivante:

  • 10 utilisateurs
  • 5 fournisseurs
  • 2 robots pour chaque utilisateur
  • 5 composantes pour chaque fournisseur, dont au moins un par type
  • 5 activités par utilisateurs, dont au moins une non débutée, une en cours et une terminée

Les options présentées dans le menu peuvent être imbriquées pour éviter d'afficher trop d'options à l'utilisateur. Cependant, le menu ne devrait pas avoir plus de 3 niveaux d'imbrication. Bien entendu, il devrait toujours être possible de retourner au menu principal sans quitter l'application.

Tâches

Pour ce troisième et dernier incrément, vous devez produire un rapport en HTML présentant votre conception détaillée (révisée) et livrer l'application complète et documentée.
Toute modification devra être mentionnée dans votre rapport. Vous pouvez vous baser sur les corrigés sommaires présentés en classe ou votre version correcte. Vous êtes tenus de fournir une solution complète du projet pour cette livraison finale.

Conception (mis à jour)

Votre diagramme de classes final et vos diagrammes de séquence doivent inclure les améliorations demandées dans les commentaires du devoir 2 et être cohérent avec le code de l'application.

Implémentation en Java

  • Application complète et robuste: Toutes les fonctionnalités sont implémentées et les entrées sont validées.
  • Conforme à la conception: Toutes les classes du diagramme de classe doivent être présentes dans le code.
  • Fichier et stockage: Les données doivent être préservées au-delà d'une session. Vous pouvez utiliser le format CSV, JSON ou XML pour persister vos données dans des fichiers texte.
  • Projet Maven: Le projet doit permettre de créer un fichier JAR (exécutable), générer la JavaDoc et exécuter les tests unitaires JUnit.

Tests unitaires avec JUnit

  • 3 tests unitaires par membre: Chaque membre doit tester une méthode qu'il n'a pas implémentée.
  • Méthodes non valides pour les tests: Constructeur, Getter et Setter, Méthode privée
  • Tous les tests implémentés doivent passer avec succès.
  • Incluez une capture d'écran de la fenêtre JUnit dans votre rapport qui montre que tous les tests unitaires passent.

Documentation

  • Manuel d'utilisation: README décrivant comment utiliser l'application
    • Titre du projet
    • Description du projet
    • Liste des fonctionnalités implémentées
    • Organisation des fichiers
    • Description des données de départ incluant le nom d'utilisateur et mot de passe d'un utilisateur par rôle
    • Instructions pour installer le projet (version Java et Maven requise, librairies externes à télécharger, etc.)
    • Instructions pour exécuter l'application
  • Documentation: Documentation du code source généré avec Javadoc.
    • Toutes vos classes, ainsi que ses attributs et ses méthodes publiques, doivent avoir une description.
    • Incluez un lien vers votre documentation JavaDoc dans votre rapport.

Remise

La remise du devoir (sur StudiUM) est un fichier ZIP ou RAR qui comprend le rapport (HTML) et tous les artefacts produits pour le projet. Le rapport doit inclure tous les éléments produits durant le projet. Téléversez toutes les images, fichiers source, fichiers de données, projet VPP et y faire référence dans le rapport.
Votre répertoire GitHub et votre remise doivent être structurés de cette façon: 

/ -- Racine du projet
├── exigences/ -- Dossier contenant les artefacts de l'exigence (diagramme de cas d'utilisation) 
├── analyse/ -- Dossier contenant les artefacts de l'analyse (diagramme d'activités) 
├── conception/ -- Dossier contenant les artefacts de la conception (diagramme de classes et diagramme de séquence) 
├── application/ -- Dossier contenant tous les artefacts de l'implémentation 
│   ├── doc/ -- Dossier contentant la documentation de l'application générée avec Javadoc 
│   ├── src/ -- Dossier contentant le code source de l'application
│   ├── test/ -- Dossier contentant les tests unitaires JUnit
│   └── robotix.jar
└── README.md
└── rapport.html

Évaluation

Ce devoir compte pour 20% de la note finale. Tout retard engendrera une pénalité de 5% par jour pour un maximum de deux jours (voir date de remise sur StudiUM).

Le devoir est à faire en équipe de trois ou quatre. Aucun devoir effectué seul ne sera accepté. Vous pouvez changer d'équipe pour ce devoir, mais c'est votre responsabilité de changer votre assignation de groupe sur StudiUM et d'en informer les auxiliaires.

Une seule personne par équipe remet le devoir complet sur StudiUM. La remise est pour le groupe au complet. Indiquez la personne qui soumet le devoir complet et le nom d'équipe.
